# Proposal

## Why

当前 Skill 商店存在两个与真实使用场景冲突的行为：

1. 远端 source 在加载阶段会对 `RegistrySkill[]` 做按 `slug` 和按 `install_name/name` 的折叠去重，导致不同 source、不同 branch、不同目录下的同名 skill 在 UI 中直接消失。
2. 本地安装状态与数据库唯一性又继续按 `name` 聚合，导致即便远端保留了两份同名 skill，用户也无法把它们作为两个独立条目安装。

这与真实用户场景不符。用户可能会：

- 在 `main` 分支维护稳定版 skill，同时在 `dev` 分支维护开发版 skill。
- 接入多个第三方商店，而这些商店恰好存在同名 skill，但内容与维护方不同。
- 需要对同名 skill 进行并行对比、灰度试用、按项目挑选、再决定是否保留某一份。

在这种情况下，PromptHub 不应该因为“名字相同”就提前隐藏另一份 skill。只有在能确认“来源身份相同且整个 skill 目录内容完全相同”时，UI 才应该做折叠或提示重复。

## Scope

- In scope:
- 重新定义 Skill 商店远端条目的身份模型，区分“显示名称”和“来源身份”。
- 取消当前按 `slug` / `install_name` 的跨来源折叠逻辑，允许同名 skill 并列展示。
- 重新定义商店卡片、详情页和已安装状态，使其按“已安装实例”而不是按名字粗暴合并。
- 设计本地 Skill 数据模型的最小扩展，使同名但不同来源/不同内容的 skill 可以独立安装。
- 设计一个对现有用户迁移成本可控的长期本地存储结构，避免因一次性重排 data 目录导致用户技能丢失、平台安装失效或路径全部变化。
- 在 UI 中增加来源标签、branch/directory 标签、以及重复/内容相同提示。
- 修复自建 Git / Gitea source 在刷新后 identity 不稳定、已导入状态消失、以及详情来源显示错误的问题。
- 明确 PromptHub 删除 skill 时对 managed repo 的生命周期语义，并修复“删除后残留 repo 内容与 sidecar 不一致”的问题。

- Out of scope:
- 本轮不设计 tag / commit SHA 锁定版本。
- 本轮不改变 Skill 文件内容格式（SKILL.md frontmatter 仍保持兼容）。
- 本轮不处理“一个本地 skill 同时映射到多个项目副本”的部署语义。

## Risks

- 当前数据库和业务层都把 `name` 视为唯一键，放开后会影响安装、更新、卸载、翻译缓存、版本快照等多条链路。
- 现有 UI 大量用 `slug` 当 React key 和详情选择键，跨 source 同 slug 时会互相覆盖。
- 过度放开后，商店可能出现大量“看起来一模一样”的卡片，必须补来源提示与内容相同提示，避免用户困惑。
- 现有用户已经在使用当前 data 目录结构；如果直接切换到完全不同的磁盘布局，会带来大规模迁移、软链接失效、第三方平台目录失配以及用户自定义脚本路径断裂风险。
- 当前删除语义与用户心智不一致：UI 上执行“删除技能”时，如果只删除数据库记录而保留 PromptHub managed repo，用户会认为删除不完整；但如果无条件删 repo，又可能误删仍被其他流程引用的容器，需要先厘清 ownership 与引用关系。
- 当前远端 source refresh 如果重新计算出不同 `source_id`，会同时导致“已导入”丢失、“看起来像新增 source/新 skill 实例”以及后续更新链路错配。

## Rollback Thinking

- 如果完全支持“同名多实例安装”改动过大，临时降级方案是：先取消商店列表去重，并在安装时强制用户填写安装别名（alias）。
- 但长期推荐方案仍然是：让系统拥有稳定的“来源身份”，并允许同名 skill 以独立实例存在，而不是把差异外包给手动改名。

## Migration Principle

- 历史用户 data 目录优先保持可读、可继续工作，而不是为了“更漂亮的新结构”做一次性大搬家。
- 新结构必须支持渐进迁移：旧目录继续可读，新安装逐步写入新布局，平台激活状态逐步迁移。
- 任何需要重命名现有 skill 目录、重写所有软链接、或要求用户手动修复路径的方案，都不应作为默认升级路径。
