# Proposal

## Why

PromptHub 目前的 Skill 管理围绕“库内 Skill”和“平台分发 Skill”展开，但真实使用中还存在一类高频场景：

- 一个本地项目仓库本身就带有 `SKILL.md` / `.claude/skills` / 其他项目级技能目录。
- 用户希望把这些项目技能纳入 PromptHub 统一查看、翻译、编辑、版本化和安全扫描。
- 这些项目技能有时需要进入 PromptHub 技能库并进一步分发到其他平台；有时则只需要作为“项目内技能”被直接管理，不应强制纳入公共技能库或平台分发流。

因此需要新增“项目级 Skill 管理”能力，让 PromptHub 能以“项目”作为新的组织维度，管理一个或多个项目目录下的 Skill。

## Goals

- 支持登记本地项目目录，并为项目定义一个或多个 Skill 扫描根目录。
- 支持发现项目级 Skill，并区分两种后续动作：
  - 纳入 PromptHub 库，再进入现有分发/平台安装流程。
  - 保持为项目级 Skill，仅在 PromptHub 中直接查看、编辑、翻译、版本化和安全扫描，不参与分发。
- 提供清晰的 UI 入口，避免项目 Skill 与“我的 Skills / 商店 / 分发”混在一起造成认知混乱。
- 尽量复用现有的 Skill 扫描预览、本地 repo 文件管理、版本快照、翻译 sidecar 与安全扫描能力。

## Non-Goals

- 本轮不做项目级 Skill 的云同步或多人协作协议。
- 本轮不做项目目录的自动 Git 绑定、分支感知或仓库状态可视化。
- 本轮不强制建立新的数据库大表来存储所有项目文件，只在需要的范围内补最小元数据持久化。
- 本轮实现项目内直接分发到项目目录的最小可用能力：默认投递到 `.agents/skills`，并允许用户额外选择多个目标目录。

## User Flows

1. 用户在左侧 Skill 区新增“项目”入口。
2. 用户添加一个本地项目目录，并可配置一个或多个 Skill 扫描目录。
3. PromptHub 扫描出项目中的 Skill 列表。
4. 用户对某个项目 Skill 可执行：
   - 直接管理：查看、编辑、翻译、版本快照、安全扫描。
   - 纳入库：复制/托管到 PromptHub Skill 库，成为普通 Skill。
   - 纳入库并分发：进入现有平台安装流程。
   - 后续需求：直接分发到当前项目的本地 Skill 目录，而不是只走全局平台分发。
5. 用户可以随时回到“项目”视图，查看不同项目下的 Skill 分布和状态。

## Risks

- 如果把“项目”做成二级筛选而不是独立入口，用户很难区分“项目内管理”和“库内 Skill”。
- 如果项目 Skill 直接复用现有 Skill 记录且不区分来源，后续可能混淆“纳入库前”和“纳入库后”的生命周期。
- 如果项目目录扫描规则过宽，容易把普通文档目录误识别为 Skill 根目录。

## Rollback

- 如果项目级 Skill UI 过重，可先退回到“设置页登记项目目录 + 扫描预览导入”的最小方案。
- 如果直接管理路径与库内 Skill 耦合过深，可保留“只纳入库”的降级路径。
