# Proposal - 公开技能命名空间与 Registry Slug 唯一性约束

## 背景与动因
当前 PromptHub 系统在将私有技能发布为公开技能时，允许不同作者发布完全同名的技能（例如，张三和李四都可以发布 `get-weather`）。
虽然这在本地存储中通过 `名称_slug__UUID` 文件夹进行了隔离，但对于公开技能市场（Marketplace）和管理员审批来说，由于没有命名空间和唯一性约束，会导致以下问题：
1. **名字抢占与混淆**：在大厅里可能会出现多个完全同名的技能卡片，导致普通用户在下载时无从区分哪个才是原作者的版本。
2. **管理员审批心智负担**：管理员无法高效判断某个申请公开的技能是否与已有公开技能冲突或属于重复提交。

为了彻底规范化公开技能的标识，我们决定采用**类似 npm 命名空间**的规范（以 `{用户名}/{技能名}` 作为唯一公开标识 `registry_slug`），并在数据库层面增加对 `registry_slug` 唯一索引的约束。

## 范围
- **在范围内 (In Scope)**:
  - 增加数据库唯一索引：在 `skills` 表的 `registry_slug` 字段上创建唯一性索引。
  - 自动生成机制：在技能发布为公开（或提交审核）时，服务端应根据当前发布者的 `username` 自动生成格式为 `{username}/{skill_name_slug}` 的 `registry_slug`。
  - 冲突校验：在管理员通过审批（`Approve`）或者普通用户提交发布请求时，如果已存在相同的 `registry_slug`（且 ID 不同），直接拦截并报错 `409 Conflict`，提示该技能标识已被占用。
  - 页面展现：在首页 Marketplace 的技能卡片或详情页中，以 `{用户名}/{技能名}` 的命名空间格式展示该技能。

- **不在范围内 (Out of Scope)**:
  - 强行修改私有技能的名称。私有技能依然在用户本地个人空间内保持原名，只有在“发布到公开市场”时，才会被赋予并校验全局唯一的 `{用户名}/{技能名}` 命名空间。

## 风险与规避
- **历史数据兼容**：老版本数据中的 `registry_slug` 可能是平铺的（如 `get-weather`）或者为空。
- **规避手段**：在数据库迁移（Migration）时，编写迁移脚本，将已有公开技能的 `registry_slug` 自动补全为 `{所有者用户名}/{技能名}`。如果所有者为空，则补全为 `system/{技能名}`。

## 回滚思考
- 如需回滚，只需在迁移脚本中删除新建的唯一索引即可。
