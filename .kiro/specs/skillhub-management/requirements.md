# 需求文档

## 引言

SkillHub 是一个用于浏览、共享和下载可复用 AI **技能（Skill）** 的管理界面（技能是以 `SKILL.md`
作为入口文件的目录级软件包）。它服务于两类用户：

- **匿名访客** 可以浏览和搜索所有**公开**技能，并下载其中任意一个。
- **已登录用户** 在此基础上还可以管理自己的**私有**技能，并将私有技能**发布**为公开技能。

本文档为 PromptHub 单体仓库（monorepo）规定该功能的行为。该功能是多用户的，且需要登录，因此其主要
归属是自托管 Web 应用（`apps/web`）——该应用中已存在一套认证系统（bcrypt + JWT、`users` /
`refresh_tokens` 表）以及服务端 `prompthub.db`。存储原语仍由 `packages/db` 拥有，共享业务逻辑由
`packages/core` 拥有，共享契约由 `packages/shared` 拥有。

### 与现有存储的关系（重要）

`packages/db/src/schema.ts` 中的现有 schema 已经提供了该功能所需的大部分内容，而项目的“单一事实来源”
规则禁止在已有表存在的情况下再创建相互竞争的表：

- `skills` 表已存储 `id`、`name`、`description`、技能路径列（`canonical_skill_path`、
  `local_repo_path`）、一个 `owner_user_id` 外键，以及一个取值约束为 `'private'` 或 `'shared'` 的
  `visibility` 列。
- `users` 表已存储账户身份信息（`id`、`username`、`password_hash`、`role`）。
- `apps/web` 中的 `refresh_tokens` 表和 `auth.service.ts` 已实现登录。

用户最初的请求描述了三张“新”表（技能基本信息表、技能-用户关系表、用户信息表）。由于等价的存储已经存在，
本规格说明将该请求视为**复用并扩展**现有的 `skills`、`users` 及所有权模型，而不是引入并行的表。具体的
数据模型决策（是扩展现有的 `skills.owner_user_id` + `visibility`，还是引入独立的关系表）作为一项开放
设计冲突记录在“开放问题与设计冲突”一节中，必须在实现前解决。本需求文档的编写方式确保其在任一决策下都
保持有效。

## 术语表

- **Skill（技能）**：一个目录级软件包，其必需入口为带有 YAML frontmatter 的 `SKILL.md` 文件。整个目录
  树（不含被忽略的条目，如 `.git` 和 `.prompthub`）构成该技能。
- **Public Skill（公开技能）**：可见性为 `shared` 的技能，任何访客都可浏览和下载。
- **Private Skill（私有技能）**：可见性为 `private` 的技能，仅其拥有者可访问。
- **Owner（拥有者）**：对某个技能而言，由 `skills.owner_user_id` 标识的已认证用户。
- **Visitor（访客）**：SkillHub 的任意客户端，无论是否已认证。
- **Authenticated_User（已认证用户）**：完成登录并持有有效会话的访客。
- **Skill_Catalog_Service（技能目录服务）**：负责在可见性规则约束下列出、过滤和搜索技能的后端组件。位于
  `packages/core`，通过 `packages/db` 访问存储。
- **Auth_Service（认证服务）**：现有的后端认证组件（`apps/web/src/services/auth.service.ts`），负责
  登录、会话签发和会话校验。
- **Skill_Download_Service（技能下载服务）**：负责将完整技能包组装为可下载归档文件的后端组件。
- **Skill_Publisher（技能发布器）**：负责将技能可见性从 `private` 转换为 `shared` 的后端组件。
- **SkillHub_UI（SkillHub 界面）**：渲染浏览、搜索、下载、登录和私有技能管理的 Web 客户端界面
  （`apps/web/src/client`）。
- **Skill_Archive（技能归档）**：包含完整技能包目录树的可下载产物（例如一个 ZIP 文件）。
- **Visibility（可见性）**：技能的存储访问分类，取值为 `private` 或 `shared` 之一。

## 需求

### 需求 1：浏览公开技能

**用户故事：** 作为访客，我希望在 SkillHub 首页浏览所有公开技能，以便我无需账户即可发现可复用的技能。

#### 验收标准

1. 当访客打开 SkillHub 首页时，技能目录服务（Skill_Catalog_Service）应在 2 秒内返回所有可见性等于 `shared` 的技能，并按技能名称升序排列。
2. 技能目录服务应将所有可见性等于 `private` 的技能排除在面向任何访客的公开浏览视图返回结果之外。
3. 当技能目录服务返回公开技能摘要时，对于每个返回的技能，技能目录服务应包含技能标识符、名称和描述，其中描述截断为 500 个字符。
4. 当没有任何技能的可见性等于 `shared` 时，SkillHub 界面应显示一条空状态消息，指明当前没有可用的公开技能。
5. 在公开技能数量超过页大小 20 的情况下，技能目录服务应分页返回结果，并应包含总数、当前页码和条目索引范围。
6. 当访客从列表中选择某个公开技能时，SkillHub 界面应显示所选技能的名称、描述以及 `SKILL.md` 内容。
7. 如果目录检索失败，则 SkillHub 界面应显示一条错误消息，并应保留先前显示的状态。
8. 如果无法检索到所选技能的 `SKILL.md` 内容，则 SkillHub 界面应显示该技能的名称和描述，并应指明内容不可用。

### 需求 2：搜索与查询公开技能

**用户故事：** 作为访客，我希望搜索和过滤公开技能，以便我能快速找到与我需求相关的技能。

#### 验收标准

1. 当访客提交一个去除首尾空白后包含 1 到 200 个字符的搜索查询时，技能目录服务应仅返回可见性等于 `shared` 且其名称或描述以不区分大小写的子串方式匹配该查询文本的技能。
2. 如果搜索查询在去除首尾空白后为空，则技能目录服务应返回所有可见性等于 `shared` 的技能的未过滤列表。
3. 如果搜索查询在去除首尾空白后超过 200 个字符，则技能目录服务应仅对去除空白后查询的前 200 个字符进行匹配，并应返回结果集而不引发错误。
4. 如果搜索查询包含 SQLite 全文搜索操作符字符，则技能目录服务应将该查询视为字面搜索文本，并应返回结果集而不引发查询语法错误。
5. 当搜索查询未匹配到任何可见性等于 `shared` 的技能时，SkillHub 界面应显示一条空状态消息，指明没有技能与该查询匹配。
6. 技能目录服务应对搜索查询执行不区分大小写的匹配。
7. 当访客提交搜索查询时，对于最多 10000 个技能的目录，技能目录服务应在 2 秒内返回匹配结果集。

### 需求 3：下载技能

**用户故事：** 作为访客，我希望下载一个公开技能，以便我能在本地安装并使用完整的技能包。

#### 验收标准

1. 当访客请求下载某个可见性等于 `shared` 的技能时，技能下载服务（Skill_Download_Service）应生成一个包含完整技能包目录树的技能归档，并应在 30 秒内为不超过最大支持大小的技能包向访客提供该技能归档。
2. 技能下载服务应在每个技能归档中包含 `SKILL.md` 入口文件。
3. 技能下载服务应从技能归档中排除被忽略的条目 `.git` 和 `.prompthub`。
4. 技能下载服务应支持为总未压缩大小不超过 500 MB（不含被忽略条目）的技能包生成技能归档。
5. 如果访客请求下载某个可见性等于 `private` 的技能且该访客不是其拥有者，则技能下载服务应拒绝该请求、不生成技能归档，并返回一个指明该技能不可访问的授权错误。
6. 如果访客请求下载一个不存在的技能标识符，则技能下载服务应返回一个指明未找到该技能标识符的未找到错误。
7. 如果技能归档在完全写入之前生成失败，则技能下载服务应中止操作、不向访客提供部分技能归档，并返回一个指明归档生成失败的错误。
8. 对于所有技能包，解压一个技能归档应重现源技能包所包含的相同文件集合及文件内容（不含被忽略条目）（往返一致性属性）。

### 需求 4：用户认证

**用户故事：** 作为已注册用户，我希望能够登录，以便我能管理自己的私有技能。

#### 验收标准

1. 当访客提交与某条已存储用户记录匹配的用户名和密码时，认证服务（Auth_Service）应在 3 秒内为该用户建立一个已认证会话。
2. 如果访客提交的凭证与任何已存储用户记录都不匹配，则认证服务应拒绝登录，并应返回一个不透露是用户名还是密码错误的“凭证无效”错误，同时保持任何现有会话状态不变。
3. 在访客持有有效已认证会话期间，SkillHub 界面应显示私有技能管理区域。
4. 在访客没有有效已认证会话期间，SkillHub 界面应隐藏私有技能管理区域，并应呈现一个登录入口。
5. 当已认证用户请求登出时，认证服务应在 2 秒内使活动会话失效，使得后续使用该会话的私有技能操作被拒绝。
6. 如果一个针对私有技能操作的请求在没有有效已认证会话的情况下到达，则技能目录服务应拒绝该请求，并应返回一个“需要认证”错误而不执行所请求的操作。
7. 如果访客针对同一用户名连续 5 次提交不匹配的凭证，则认证服务应在 15 分钟内拒绝该用户名的所有后续登录尝试，并返回一个“账户暂时锁定”错误。
8. 当一个已认证会话在 30 分钟内没有任何已认证请求时，认证服务应使该会话失效，从而使下一个请求被视为没有有效已认证会话。
9. 如果访客提交的用户名超过 254 个字符或密码超过 128 个字符，则认证服务应拒绝登录，并应返回一个校验错误而不建立会话。

### 需求 5：管理私有技能

**用户故事：** 作为已认证用户，我希望查看我自己的所有私有技能，以便我能管理尚未公开共享的技能。

#### 验收标准

1. 当已认证用户打开私有技能管理区域时，技能目录服务应在 2 秒内返回所有 `owner_user_id` 等于发起请求用户标识符且可见性等于 `private` 的技能。
2. 在返回给发起请求的已认证用户的私有技能结果中，技能目录服务应返回正好零个由其他用户拥有的技能。
3. 当技能目录服务返回私有技能摘要时，对于每个返回的技能，技能目录服务应包含技能标识符、名称、描述和可见性。
4. 当已认证用户选择其某个私有技能时，SkillHub 界面应显示该技能的名称、描述以及 `SKILL.md` 内容。
5. 当私有技能管理区域为某个已认证用户返回零个结果时，SkillHub 界面应显示一条空状态消息，指明不存在私有技能。
6. 如果一个未认证请求针对私有技能管理区域，则技能目录服务应拒绝该请求、返回一个“需要认证”错误，并且不返回任何技能数据。
7. 如果私有技能检索失败，则 SkillHub 界面应显示一条错误消息、不显示部分结果，并保留先前状态。
8. 如果无法加载所选技能的 `SKILL.md` 内容，则 SkillHub 界面应显示该技能的名称和描述，并应指明内容不可用。

### 需求 6：发布私有技能

**用户故事：** 作为已认证用户，我希望发布我的某个私有技能，使其成为所有访客都可发现和下载的公开技能。

#### 验收标准

1. 当已认证用户请求发布其拥有的某个可见性等于 `private` 的技能时，技能发布器（Skill_Publisher）应将该技能的可见性设为 `shared`、完成发布，并在 5 秒内返回一个成功结果。
2. 当技能发布器将某个技能的可见性设为 `shared` 时，技能目录服务应在 5 秒内将该技能纳入公开浏览和搜索结果。
3. 如果已认证用户请求发布其不拥有的技能，则技能发布器应拒绝该请求、保持存储的可见性不变，并返回一个指明非拥有者的授权错误。
4. 如果已认证用户请求发布其可见性已等于 `shared` 的技能，则技能发布器应返回一个指明该技能已公开的结果，并保持存储的可见性不变。
5. 当技能发布器更改某个技能的可见性时，技能发布器应保持该技能的 `owner_user_id`、标识符、名称、描述和技能包内容不变。
6. 当技能发布器完成一次发布时，技能发布器应在单个数据库事务内将更新后的可见性持久化到 `skills` 表。
7. 如果已认证用户请求发布的目标技能标识符不存在，则技能发布器应拒绝该请求、不做任何数据更改，并返回一个未找到错误。
8. 如果发布的数据库事务失败，则技能发布器应回滚该事务、将技能的可见性保持为 `private`，并返回一个失败错误。

### 需求 7：技能数据模型与可见性存储

**用户故事：** 作为维护者，我希望技能的所有权和可见性在现有数据库中被一致地存储，以便公开/私有行为可靠且不重复存储状态。

#### 验收标准

1. 技能目录服务应将 `skills` 表视为技能标识符、名称、描述和技能路径元数据的单一事实来源。
2. 技能目录服务应将 `skills.visibility` 视为判断技能是公开（`shared`）还是私有（`private`）的单一事实来源。
3. 技能目录服务应将 `skills.owner_user_id` 视为判断技能由哪个用户拥有的单一事实来源。
4. 认证服务应将 `users` 表视为用户账户身份的单一事实来源。
5. 在某条技能记录的 `owner_user_id` 为空的情况下，技能目录服务应将该技能排除在每个用户的私有技能结果之外。
6. 在持久化可见性变更时，技能发布器应在 `skills.visibility` 中存储 `private` 或 `shared` 二者之一。
7. 如果一次可见性写入收到 `private` 或 `shared` 以外的值，则技能发布器应拒绝该写入、保留现有存储值，并返回一个错误。
8. 在某条技能记录的可见性值为空或无法识别的情况下，技能目录服务应将该技能视为 `private`，并将其排除在公开结果之外。
9. 在所采用的数据模型引入了专用的技能-用户关系表的情况下，该关系表应通过 `ON DELETE` 行为为 CASCADE 或 SET NULL 的外键引用 `skills.id` 和 `users.id`，并且每当 `owner_user_id` 发生变化时，技能目录服务应在同一事务内将该表与 `skills.owner_user_id` 协调一致。

### 需求 8：可见性与访问控制强制

**用户故事：** 作为用户，我希望私有技能保持私有，使我尚未发布的工作成果不暴露给他人。

#### 验收标准

1. 当某个可见性等于 `private` 的技能的拥有者请求该技能时，技能目录服务应返回该技能；同时，技能目录服务不应向除其拥有者之外的任何请求者返回可见性等于 `private` 的技能。
2. 如果已认证用户请求由其他用户拥有的私有技能，则技能目录服务应返回一个未找到或授权错误，并且不应返回该技能的内容。
3. 技能目录服务应向任何访客返回可见性等于 `shared` 的技能，无论其认证状态如何。
4. 当访客请求技能的列表或搜索时，技能目录服务应从结果中排除每一个可见性等于 `private` 且其拥有者不是发起请求者的技能。
5. 如果一个未认证请求针对某个私有技能管理操作，则技能目录服务应返回一个“需要认证”错误，并且不应执行该操作。
6. 当技能目录服务收到技能标识符或搜索输入时，在执行任何数据库查询之前，技能目录服务应校验：技能标识符匹配其定义的标识符格式，且搜索输入长度介于 0 到 256 个字符之间并且不包含空字节或控制字符。
7. 如果传入的技能标识符或搜索输入因超过 256 个字符、包含空字节或控制字符、或不匹配定义的标识符格式而未通过校验，则技能目录服务应以校验错误拒绝该请求、不执行数据库查询，并且不返回任何技能内容。

### 需求 9：SkillHub 界面的国际化

**用户故事：** 作为任意受支持语言环境下的用户，我希望 SkillHub 文本以我的语言显示，使该功能在所有受支持语言环境下都可用。

#### 验收标准

1. SkillHub 界面应通过 i18n 翻译函数渲染所有面向用户的文本，包括按钮标签、错误消息、输入占位符、工具提示和状态文本。
2. 在为 SkillHub 引入一个新的面向用户的字符串的情况下，该功能应在全部七个受支持语言环境文件（`en`、`zh`、`zh-TW`、`ja`、`fr`、`de`、`es`）中提供翻译键，且这七个文件中的每一个都应为该键包含一个非空的翻译值。
3. SkillHub 界面不应在语言环境资源文件之外包含硬编码的面向用户的显示字符串，包括硬编码的中文字符。
4. 当 SkillHub 界面显示包含一个或多个动态值的文本时，SkillHub 界面应使用 i18n 插值来组合文本，而不是拼接已翻译的字符串片段。
5. 如果某个请求的翻译键在当前激活语言环境中缺失或解析为空值，则 SkillHub 界面应渲染默认语言环境（`en`）的值，并且不应显示原始翻译键或空字符串。

## 开放问题与设计冲突

以下事项必须在设计阶段解决（依据项目的“设计冲突停止规则”），因为它们会改变数据模型或宿主应用：

1. **宿主应用**：本规格说明假设这个多用户、基于登录的 SkillHub 位于自托管 Web 应用（`apps/web`）中，
   因为桌面端 `prompthub.db` 是本地优先且单用户的。请确认 SkillHub 是否还必须出现在 Electron 桌面端
   渲染进程中——若是，则需要在 `packages/shared/constants/ipc-channels.ts` 中建立 渲染进程 → preload →
   主进程 的 IPC 通道。
2. **所有权数据模型**：用户最初的请求要求一张独立的技能-用户关系表，但现有 `skills` 表已经携带
