# Requirements: demo-app

> 更新时间：2026-05-26
>
> 这份文档只写 what / why，不写 how。

## 1. 需求摘要

[交付一个以浏览器页面为主要入口的产品，让用户能在关键页面完成核心流程，并清楚看到当前状态和下一步操作。]

## 2. 用户故事

- `US-001` 作为首次访问用户，我希望快速理解页面主价值并完成首个核心操作，以便判断这个产品是否值得继续使用
- `US-002` 作为回访用户，我希望在界面中查看状态、结果和错误提示，以便继续推进我的任务

## 3. 功能需求

| ID | 描述 | 为什么需要 | 优先级 |
|---|---|---|---|
| `FR-001` | 用户能够在关键页面中完成首个核心流程 | 降低首次使用门槛，减少流失 | H |
| `FR-002` | 用户能够在界面中看到当前状态、错误提示和下一步操作 | 降低失败成本，提升可恢复性 | H |

## 4. 非功能需求

| ID | 描述 | 度量方式 |
|---|---|---|
| `NFR-001` | 关键页面在目标网络条件下应在 3 秒内可交互 | 使用浏览器性能指标或 Web Vitals 测量 |
| `NFR-002` | 页面错误提示不得暴露内部实现和敏感信息 | 走查错误展示内容与日志策略 |

## 5. 验收标准

| ID | 对应需求 | 验收条件 |
|---|---|---|
| `AC-001` | `FR-001` | 当用户进入关键页面时，应能在不阅读额外文档的情况下开始首个核心流程 |
| `AC-002` | `FR-002` | 当请求失败或校验失败时，界面应显示明确、可恢复的提示，并保留必要上下文 |

## 6. 优先级与版本边界

| Requirement | 优先级 | 目标版本 | 说明 |
|---|---|---|---|
| `FR-001` | H | V1 | 没有首个可完成流程，页面项目无法验证价值 |
| `FR-002` | H | V1 | 状态和错误反馈直接影响可用性 |

## 7. 范围外内容

- 原生移动端应用
- 复杂运营后台或多角色权限系统

## 8. 外部依赖与约束

- 浏览器运行环境
- 后端 API 或 BFF
- 设计规范、组件规范或品牌约束

## 8.1 新手填写提示

如果你还不知道怎么写，可以先回答下面这些问题：

| 问题 | 为什么要先确认 |
|---|---|
| 哪个页面是 V1 最重要的关键页面？ | 没有关键页面，就无法聚焦首个可完成流程 |
| 主要用户更常在手机还是桌面使用？ | 会直接影响体验目标和后续设计规范 |
| 是否需要登录、权限或多角色？ | 会影响页面流程和需求范围 |
| 是否已有品牌色、字号或组件规范？ | 会影响设计阶段的视觉和组件约定 |
| 必须支持哪些浏览器和最低设备能力？ | 会影响性能与兼容性边界 |

## 8.2 可直接参考的 V1 示例

如果你做的是常见后台或工具型 Web，可以先参考：

- V1 关键页面：登录后首页或列表页
- V1 核心流程：创建一条记录、查看状态、修改一次关键字段
- V1 必要反馈：加载中、成功、失败、空状态
- V2 候选内容：多角色权限、复杂筛选报表、批量操作、主题换肤

## 9. 需求自检

- 每条 `FR-*` 都能被单独验证
- 每条高优需求至少对应一个 `AC-*`
- 这里没有写技术栈、数据库、组件划分、接口实现细节
- 读完这份文档，不看代码也知道“项目要做什么”

## 10. 待确认

- 关键页面的首个核心流程具体是什么
- 是否存在登录、权限或多角色场景
- 需要支持的浏览器范围和最低设备能力
- 是否已有品牌、字体、组件库或设计系统约束

## 10.1 常见错误示例

错误写法：

- “首页用 React，列表用 TanStack Table，表单用某某组件库。”

更好的写法：

- “用户可以在关键页面完成创建、查看和修改的主流程。”

## 不该写进这里的内容

- 技术栈名称
- 表结构和字段设计
- 类名、函数名、文件名
- API 路径细节
- 组件拆分方案
