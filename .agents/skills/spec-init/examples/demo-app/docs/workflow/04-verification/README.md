# Verification Plan: demo-app

> 更新时间：2026-05-26
>
> 先定义如何证明正确，再开始大规模实现。

## 1. 测试目标

- 优先防住页面关键流程不可完成、状态丢失和错误提示失效
- 优先覆盖首次访问用户和回访用户的关键浏览器路径

## 2. 测试层次

| 层次 | 目标 | 覆盖内容 | 工具 / 方式 |
|---|---|---|---|
| White-box Unit | 验证页面局部逻辑 | 输入校验、状态派生、展示条件、关键分支 | [待定，例如组件测试或纯逻辑测试] |
| Integration | 验证页面与数据层协作 | 表单提交、异步请求、错误反馈 | [待定，例如页面集成测试] |
| E2E | 验证关键用户流程 | 浏览器中的首个核心流程和失败恢复 | [待定，例如浏览器自动化测试] |
| Performance | 验证性能目标 | 首屏交互时间、关键操作响应、长列表或大数据场景 | [待定，例如性能基准或 Web Vitals] |
| Security | 验证安全边界 | 鉴权、权限、输入校验、敏感信息展示 | [待定，例如安全负向测试] |

## 3. 需求到测试映射

| Test ID | Requirement | Test Type | 核心断言 |
|---|---|---|---|
| `TEST-001` | `FR-001` | E2E | 用户可以在关键页面完成首个核心流程 |
| `TEST-002` | `FR-002` | Integration | 请求或校验失败时，页面会显示可恢复的反馈 |

要求：

- 每个高优先级 `FR-*` 至少映射到一种自动化测试
- 与当前版本相关的白盒、性能、安全测试必须进入计划
- 如果某条需求暂时不能自动化验证，要写清原因和替代验证方式

## 4. 第一批红灯测试

1. 访问关键页面后，首次用户无法找到首个操作入口
2. 提交关键表单后，页面没有显示加载中和最终结果
3. 请求失败后，错误提示不可见或无法恢复上下文

## 5. 测试数据与夹具

- 准备成功、失败、空态三类页面数据样例
- 为关键浏览器流程准备稳定账号、种子数据或 API mock

## 5.1 白盒测试重点

- 组件内部状态分支
- 表单校验逻辑
- 页面错误反馈与状态切换逻辑

## 5.2 性能测试计划

- 关键性能指标：首屏可交互时间、关键提交响应时间
- 压测场景：高数据量列表、连续交互、慢网络
- 通过标准：[例如关键页面 p95 在目标范围内]

## 5.3 安全测试计划

- 鉴权和权限边界
- 输入校验和注入风险
- 敏感信息是否误显示到界面或日志

## 6. 回归策略

- 关键页面交互流必须保留浏览器回归测试
- 任何与状态管理、表单校验、错误提示相关的 bug fix 后都要补回归测试
- 影响路由、权限或 API 交互的改动需要完整回归关键路径

## 7. 测试执行建议

- 先从首个关键页面的失败测试开始
- 优先覆盖交互密集、状态切换多、容易返工的路径
- 如果项目还没选定测试框架，先写测试思路和断言，再延后具体工具选择
- 不要先写实现再回头补测试；页面行为边界要由测试先约束

## 8. 完成定义

- 每个高优先级需求至少有一种自动化验证
- 关键路径的失败场景有覆盖
- bug 修复必须补回归测试
- 测试命令可以被稳定重复执行
- 白盒、性能、安全要求已被显式检查或标明延期原因
