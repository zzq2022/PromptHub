# Testing Standards

## 基本原则

- 高优先级需求必须有自动化验证。
- 先写失败测试，再进入主要实现。
- bug 修复必须补回归测试。
- 测试必须服务真实风险，不允许为了“看起来有测试”而瞎写。

## 测试分层

- White-box Unit: 验证纯逻辑、转换规则、边界条件和关键分支
- Integration: 验证模块、接口和依赖协作
- E2E: 验证最关键的用户流程或调用路径
- Performance: 验证性能目标和高负载表现
- Security: 验证鉴权、权限、输入校验和敏感信息保护

## 变更要求

- 修改需求时，先检查 `docs/workflow/04-verification/README.md` 是否需要同步更新。
- 修改外部契约时，优先补集成或端到端测试。
- 引入新风险路径时，要明确写入回归策略。
- 对关键路径，至少说明白盒、性能、安全测试是否适用；适用时必须落地，不适用时也要写明原因。
