# Clarification Rules

## 基本原则

- 对需求边界、目标用户、成功标准、非目标、技术栈、架构方向、数据模型、权限模型、测试覆盖策略存在关键疑点时，必须先问用户。
- 不要把“我觉得合理”当成“用户已经确认”。
- 可以给用户方案选择，但必须讲清楚利弊、风险和推荐理由，不能直接替用户定案。

## 必须确认的场景

- 不同选择会改变需求范围或产品行为
- 不同技术方案会明显影响架构方向、维护成本或交付速度
- 不同数据模型或权限模型会影响后续兼容性
- 不同测试范围会影响质量保证边界
- 覆盖已有文件、迁移旧数据、替换已有实现等不可逆操作

## 按项目类型补充必问问题

- Web 项目：关键页面是什么、主要终端是手机还是桌面、支持哪些浏览器、是否已有品牌 / 设计系统、是否有登录或多角色场景。
- API / Service 项目：首批资源和动作是什么、调用方是谁、认证方式是什么、是否需要分页 / 幂等 / 批量 / webhook、是否存在审计或合规要求。
- CLI 项目：第一个必须工作的命令是什么、目标操作系统和 shell 是什么、主要给人还是给自动化脚本使用、是否需要 JSON 输出、是否有破坏性操作确认。

如果用户暂时答不全，也要把这些问题写进 `docs/workflow/00-intake/README.md` 或 `docs/workflow/01-requirements/README.md` 的待确认部分，而不是在实现阶段再猜。

## 推荐提问格式

1. 先说明当前疑点是什么
2. 再给 2 到 3 个候选方案
3. 对每个方案写清：适用场景、优点、代价、风险
4. 如果你有建议，可以给出推荐，但必须明确“这是建议，不是已确认决定”

## 禁止行为

- 因为省事而跳过关键澄清
- 在关键决策未确认时，默认采用自己偏好的技术栈或架构
- 用模糊措辞掩盖“其实没有确认过”的事实
